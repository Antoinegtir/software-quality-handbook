# Software Quality Handbook

A practical guide to help engineers at our startup work consistently, ship faster, and reduce defects in production.

## Purpose

As our team grows, having each engineer work differently creates unpredictable progress and too many bugs reaching production. This handbook defines **the right way to do things here**, not rigid rules, but proven practices that help us move fast without breaking things.

## Contributing

We follow **trunk-based development**:

1. Create a branch from `main`
2. Make your changes
3. Open a Pull Request
4. Get at least one review from a teammate
5. Merge only after approval

Never commit directly to `main`.
